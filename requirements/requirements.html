  1. Analyses function exception handling Description: Low level analysis functions shall pass all exceptions to the calling function to be handled.

  2. Analyses file location Description: Each type of analyses shall have a single file in the analyses directory such that it can be imported as from ramstk.analyses import [name of analysis]. This file may contain functions, classes, or both as necessary to implement the analysis. Rationale: analyses/MilHdbk217f.py contains the class MilHdbk217F. This class contains methods to perform the MIL-HDBK-217F parts count and parts stress analyses.

  3. Analyses model location Description: For analyses that rely on engineering or mathematical models and/or constants specific to each component type, the associated models and constants shall reside in a single file for each component type in the analyses/models/[analysis name] directory. Rationale: The MIL-HDBK-217F models for capacitors reside in analyses/models/milhdbk217f/Capacitor.py.

  4. Return values when value is not used. Description: In some instances not all component subcategories will use every intermediate value. For methods/functions that calculate intermediate values, return a 0.0 as the default value. Rationale: Rather than littering the code with if-then structures to only calculate the needed intermediate values, just call all the functions and return 0.0 for the unused values. They’re unused, so they won’t affect the final output anyway. For example, in MIL-HDBK-217F calculations for semiconductor devices, there is no electrical stress (piS) for diodes. Rather than putting an if(subcategory != 2) statement to prevent calculating piS, just calculate the piS value and then don’t use it. In the function for calculating piS, if there’s an if-then construct based on subcategory ID, end it with a default else: piS = 0.0.

  5. GUI element callback naming criteria Description: GUI element callback method/function shall be named send_request_do_[action]. Rationale: This naming convention represents what a user is doing when interacting with the application. For example, a user wants to calculate a component’s hazard rate. She would press the Calculate button whose callback method would be named send_request_do_calculate_hardware.

  6. Data controller method naming criteria Description: Data controller methods shall be named request_do_[action]. Rationale: The data controller’s function is to coordinate the wishes of the user and the actions of the models. The data controller listens for requested actions from the user (via the GUI) and directs the data model to perform the requested action. For example, request_do_calculate_hardware would be called in response to the send_request_do_calculate_hardware from the GUI.

  7. Data model method naming criteria Description: Data model methods shall be named do_[action] Rationale: Data models are the workhorses of the application. This naming convention accurately describes that because data models “do” things. For example, the data controller request_do_calculate_hardware method would call the Hardware data model’s do_calculate method.

  8. GUI published message naming criteria Description: Messages published by the GUI shall be named ‘request_do_[action]’. Rationale: The GUI is the interface for the user to perform the actions the user desires. It is through the GUI the user “requests” these actions to be perofrmed.

  9. Data model published message naming criteria on success Description: Messages published by the data model shall be named succeed_[action] when the requested action was successfully completed. Rationale: This forms the closure of the loop between the requestor (GUI), manager (data controller), supervisor (data model), worker (method/function).

  12. Handle exceptions at the level that knows how to handle them Description: Exceptions shall be handled at the lowest level there is sufficient information to handle the exception. Rationale:* Exceptions need to be handled at the level that can produce an informative error log message as well as an informative error, warning, or information dialog for the user.

  13. Do not handle programming exceptions Description: Exceptions such as IndexError, TypeError, KeyError, and NameError are programming errors and should not be handled by RAMSTK, they should be handled by the programmer. Rationale: Handling programming exceptions simply hides bugs.

  14. Document the exceptions thrown by a function/method Description: Add :raise: directives to the docstrings of functions and methods to describe all the exceptions that function/method could raise. This includes programming exceptions. Each exception should briefly describe what condition(s) will throw that exception. Rationale: Documenting raised exceptions ensures the programmer of client functions/methods prevents passing bad inputs and is aware of the possible need to handle a run-time/user exception in their code.

  15. Dict keys start with one (1) Description: When using integer dict keys, start with one (1). Rationale: Integer-keyed dicts are often associated with input variables whose value is the index in a combobox. Since comboboxes always contain a blank in position zero, starting the dict index at 1 provides a 1:1 correlation.

  16. Trailing comma only for singletons Description: Only singleton lists and tuples should include the trailing comma. Rationale: This is a preference.
